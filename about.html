What this is

A federal US sister to House (UK) and House (AU), built for Guardian US journalists who need to triage Congress faster than the official tools allow. Three surfaces planned, in the same shape AU settled on: Search across the Congressional Record, bills and votes; Deep Dive for plotting a term over time as a party-stacked timeline; and Lobbying for searching Lobbying Disclosure Act filings by registrant, client, lobbyist and the free-text issue-description field. The third is the headline US surface — UK and AU have no equivalent because their lobbying registers are pitiful by comparison.

Use cases that drove the shape: a politics correspondent reading every mention of a bill across both chambers in a sitting week; an investigations reporter tracking who lobbied on a specific rule the morning after the quarterly LD-2 batch drops; a tech reporter scanning a hearing’s witness statements before the gavel falls; a Washington editor following a single member’s floor speeches across a year.

How it’s being made

Vanilla JavaScript on GitHub Pages, mostly. The US sits between UK and AU on the data spectrum, so the architecture is hybrid, and that’s an honest description rather than a fudge. The daily-news layer is UK-shaped: bills, votes, members and Congressional Record metadata come from api.congress.gov; the Federal Register from federalregister.gov; campaign finance from api.open.fec.gov. All CORS-friendly, all free, all called from a small Cloudflare Worker that injects API keys and smooths burst traffic.

The accountability layer is AU-shaped, because the source resists. Lobbying Disclosure filings live at lda.gov — REST API exists, but no CORS and token auth, so a nightly GitHub Action harvests filings into static JSON shards and the browser searches those client-side. Same pipeline pattern as House AU’s Hansard and Estimates harvesters, different parser. Quarterly cadence: LD-2 filings drop ~20 days after each quarter ends, so four times a year the shard set updates in a flood and the rest of the time the harvester is mostly idle.

Built with Claude Code, internal newsroom tool, currently hosted on my personal GitHub. US federal government works are public domain — materially cleaner than UK’s Open Parliament Licence and miles cleaner than AU’s CC-BY-NC-ND. The expectation is it moves to the Guardian stack behind authentication once it’s doing useful work.

What it doesn’t do

It doesn’t mirror UK House’s Committees and Transcripts surface. The official US equivalent — GovInfo’s CHRG collection — publishes the finalised hearing record six to eighteen months after the hearing, which is useless for deadline journalism. The fresh witness statements live on per-committee sites with no aggregator and inconsistent URL patterns. Building a dedicated Hearings tab would be the same mistake AU made with Committee Transcripts before cutting it on purpose. A modest hearings feed inside Search may earn its place; a tab won’t.

It doesn’t do FEC as a primary tab. The data’s strong but the market’s saturated — OpenSecrets, FollowTheMoney and the FEC’s own search all do FEC, and reskinning what they already do unlocks nothing. FEC sits as a cross-reference layer on top of Lobbying instead.

It doesn’t do Federal Register as a primary tab either. federalregister.gov’s own UI is genuinely good. Same logic — cross-reference, not headline.

Also not in here: state-level coverage (federal first; widen the scope only if the narrow version actually gets used), C-SPAN transcripts (no public API, ToS forbids commercial redistribution, journalism use is grey-zone), Senate roll-call votes (currently blocked by Akamai for scripted clients — waiting for api.congress.gov to ship them rather than fighting the WAF), and any analogue of Senate Estimates (Congress doesn’t have one; Lobbying is the better quarterly-flood surface anyway).

Diary

May 2026

Search v0 — first real queries land

Search now talks to GovInfo through the Worker and renders ranked results: title, date, collection tag, click-through to the full document on govinfo.gov. Default collection is the Congressional Record (CREC); Bills, Hearings, the Federal Register and Committee Reports are pill toggles you can layer on. Date-range filter runs last month / last year (default) / last 5 years / custom. The search state lives in the URL, so a result page is shareable and the back button works.

One honest gap to flag: GovInfo’s search returns titles only, not the surrounding sentence. CREC titles are also shouty all-caps (“CLIMATE CHANGE”) so there’s a small de-shouting pass on the way in. Snippets-around-the-match comes next, via lazy-fetching the granule HTML for visible results with bounded concurrency. Until then, click-through is one tap away.

May 2026

Cloudflare Worker — load-bearing, not optional

Started this commit thinking the Worker was overkill for v0 and we could call the APIs direct from the browser. Pushed back on my own plan, got told to skip it, then verified actual CORS headers and discovered api.govinfo.gov sends none. Without a proxy, full-text search of the Congressional Record from a static page is impossible. So the Worker is back — same shape as House UK’s, ~50 lines, with three differences for US: it handles POST (GovInfo Search is POST), it injects the api.data.gov key server-side, and the allow-list is the four US hosts we’ll touch. Edge-caches GET responses for five minutes. Lesson: verify CORS before deciding the architecture, not after.

May 2026

Day one — static shell, brand swap, scaffold notice

Lifted the static shell, type stack, palette and component vocabulary straight from House AU. Same Source Serif 4 / Inter / JetBrains Mono, same cream paper, same near-black ink, same restrained-accent discipline. Swapped the AU eucalypt green (#00843D) for Federal navy (#1d2f5e), updated the region tag to US in the same mono-caps treatment, and dropped in the favicon, touch icon and OpenGraph card from the brand kit that already lived in regions/us/ in the AU repo. Total brand-swap cost: a single CSS-variable override and four file copies.

The homepage is honest about where it sits. Search is visible but inert; on submit it tells you the wiring lands next and points at this page for the plan. No coming-soon stubs, no broken nav links, no placeholder pages with “under construction” banners. Just a real surface that says what it is.

May 2026

Feasibility memo — agreed the shape before lifting a finger

Spent a session researching what the US data landscape actually offers in 2026 (not 2020), pushed back on a few assumptions in the brief, and settled on three surfaces matching AU’s discipline: Search, Deep Dive, Lobbying. Killer surface is Lobbying Disclosure — quarterly LD-2 floods, genuinely US-distinctive, free-text issue fields that no existing tool searches well. Hearings becomes a feed, not a tab. FEC and Federal Register become cross-references, not tabs. ProPublica’s Congress API and OpenSecrets’ API are both dead now — both got struck off the plan. The full memo and the argument behind each call lives in the project notes.

The single most important constraint to carry into the build: lda.senate.gov sunsets on 30 June 2026, with the successor at lda.gov live now. Any harvester I write points at lda.gov from day one. The legacy host is borrowed time and would be throwaway code.
